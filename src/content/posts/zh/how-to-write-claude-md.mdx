使用 Claude Code 以来，我一直没怎么用 CLAUDE.md。用过 `/init` 来初始化，也试过直接留空。我知道 CLAUDE.md 应该很重要，但始终没找到使用它的窍门。

最近看了不少长期使用 Claude Code 的人的分享后，我才想明白：CLAUDE.md 本质上是一个管理上下文的工具，用来有效地引导 LLM，而且它是**渐进演化**出来的——不需要一开始就写好。在项目初期，CLAUDE.md 完全可以为空。我之前通过 plan.md 来管理上下文，也得到了满意的效果。想通这一点后，我就放下了"一定要在 CLAUDE.md 里写点什么"的执念。

以下是几位实践者的做法，我从中学到了不少。

**[Ben's Builds](https://www.bensbites.com/p/bens-builds-2)**：他在 AGENTS.md 里写了自己的工作流、对结果的要求、LLM 应该遵循的工作方式，甚至还加了一条让 LLM 表明 AGENTS.md 已成功载入的提示（挺有意思的）。

**[OneV's Den](https://onevcat.com/2025/08/claude-code/#%E5%B0%8F%E6%AD%A5%E8%BF%AD%E4%BB%A3%E8%BF%98%E6%98%AF%E6%94%BE%E9%A3%9E%E8%87%AA%E6%88%91)**：他在 CLAUDE.md 中只列了项目的编译命令、测试命令和 linter 配置，内容很精简。不过文中提到的很多管理上下文的思路都值得借鉴。

**[Claude Code 团队的 CLAUDE.md](https://x.com/milesdeutscher/status/2034658673738580442)**：里面写的是"什么情况下应该怎么做"——通用的工作方法及其目的。本质上是一个上下文管理的规则集合，包含了过去的错误教训、惯例和约束。比如"验证通过前才算做完"，这一点很重要，因为 LLM 确实容易偷懒。

碰巧今天 Claude Code 更新了 `/init`。我试了一下，生成的 CLAUDE.md 比之前简洁了很多，只围绕几个核心问题：项目是个人还是团队的？你是什么角色？你对项目的熟悉程度如何？你希望 LLM 如何与你交流？

这让我更加确信：CLAUDE.md 不是一份需要精心设计的文档，而是在使用过程中逐渐沉淀下来的。之后我会继续学习他人管理上下文的方式，把有用的借鉴过来。但最重要的，还是放下对工具的纠结，专注于构建项目本身。
